PROJECT ARCHITECTURE AND EXECUTION FLOW

1. Purpose of the system

This project implements a geo-distributed Retrieval-Augmented Generation (RAG) pipeline split across three machines:
- Node C at the university edge
- Node B at home for dense retrieval
- Node A at home for generation and fusion

The design goal is to reduce time-to-first-token (TTFT) by overlapping retrieval work across machines instead of executing the entire pipeline sequentially on one host.
The project also serves as a benchmark for comparing a sequential baseline against a parallelized edge-orchestrated version.

At a high level, the system does this:
- Node C receives a user query.
- Node C performs local sparse retrieval with BM25 using Tantivy.
- Node C sends a dense retrieval dispatch to Node B over gRPC.
- Node B computes dense embeddings and searches Qdrant.
- Node B forwards dense results to Node A over gRPC.
- Node A waits for sparse and dense context, fuses them, and generates text with vLLM.
- Node C streams the answer back to the user.

2. Hardware and network roles

Node C (10.8.0.2)
- Runs the FastAPI gateway that users talk to.
- Hosts the local Tantivy BM25 index.
- Orchestrates the pipeline and records latency.
- Two Node C variants exist:
  - Node C/ = parallel version
  - Serial/Node C/ = sequential baseline version

Node B (10.8.0.5)
- Runs the dense retrieval worker.
- Uses BGE-M3 to embed the query.
- Uses Qdrant to perform vector search.
- Forwards the dense results to Node A.
- Exposes a gRPC dispatch service, not a synchronous retrieval service.

Node A (10.8.0.1)
- Runs the generation engine with vLLM.
- Receives sparse context from Node C.
- Receives dense context forwarded from Node B.
- Fuses both contexts using Reciprocal Rank Fusion (RRF).
- Generates the final response and streams tokens back.

Network topology
- Node C reaches Node A and Node B over WireGuard.
- Node A and Node B are also reachable from each other over the home network.
- The current working topology uses WireGuard IPs:
  - Node A: 10.8.0.1
  - Node B: 10.8.0.5
  - Node C: 10.8.0.2

3. Major communication channels

There are three distinct communication patterns in the system.

3.1 Node C to Node B
- Protocol: gRPC unary dispatch
- Purpose: tell Node B to start dense retrieval asynchronously
- RPC: DenseDispatcher.Dispatch()
- Request message: DenseDispatchRequest
- Response message: DenseDispatchAck

DenseDispatchRequest carries:
- query_id
- query_text
- top_k
- node_a_lan_host
- node_a_grpc_port

This is intentionally fire-and-forget. Node C does not wait for dense retrieval to finish before moving on.

3.2 Node B to Node A
- Protocol: gRPC unary forwarding call
- Purpose: send dense results directly to Node A after retrieval completes
- RPC: ResultForwarder.ForwardDenseResults()
- Request message: DenseResultForward
- Response message: DenseResultAck

DenseResultForward carries:
- query_id
- repeated RetrievedDocument docs
- t_dense_ms

3.3 Node C to Node A
- Protocol: gRPC bidirectional streaming
- Purpose: send sparse context and receive generated tokens back
- RPC: GenerationOrchestrator.GenerateStream()
- Request stream message: SparseContextRequest
- Response stream message: GenerationToken

SparseContextRequest carries:
- query_id
- query_text
- repeated RetrievedDocument docs
- t_sparse_ms
- node_b_dispatch_failed

GenerationToken carries:
- query_id
- token
- is_final
- ttft_ms

4. The data model

Sparse retrieval documents
- Node C BM25 returns ranked documents containing doc_id, text, score, and rank.
- These are wrapped into the SparseContextRequest docs field.

Dense retrieval documents
- Node B Qdrant results are converted into RetrievedDocument messages.
- The result includes doc_id, text, score, and rank.
- Node B forwards these to Node A.

Final fused context
- Node A collects sparse docs from Node C.
- Node A collects dense docs forwarded by Node B.
- Node A fuses both ranked lists with Reciprocal Rank Fusion.
- Node A reads the full text for the fused doc_ids from its local SQLite corpus.
- Node A builds a prompt from the fused context and the user query.

6. End-to-end step-by-step execution flow

6.1 Query arrival at Node C
1. A user sends a query to Node C’s HTTP gateway.
  Note: File: Node C/app.py. Function: query_endpoint() in the FastAPI route handler.
2. FastAPI receives the POST /query request.
  Note: File: Node C/app.py. Function: query_endpoint().
3. Node C records the request start time for TTFT measurement.
  Note: File: Node C/app.py. Function: LatencyRecorder.start() and handle_query().
4. Node C creates a query_id for correlation across all three nodes.
  Note: File: Node C/app.py. Function: handle_query().

6.2 Sparse retrieval on Node C
1. Node C immediately runs BM25 locally through Tantivy.
  Note: File: Node C/bm25.py. Function: TantivyBM25Index.search().
2. The sparse retrieval returns ranked documents from the local index.
  Note: File: Node C/bm25.py. Function: TantivyBM25Index.search().
3. Node C stores the sparse latency as t_sparse_ms.
  Note: File: Node C/app.py. Function: handle_query() and LatencyRecorder.finish().
4. The sparse documents are prepared for Node A as RetrievedDocument messages.
  Note: File: Node C/app.py. Function: handle_query() using coordination_pb2.RetrievedDocument.

6.3 Dense retrieval dispatch to Node B
1. Node C sends a DenseDispatchRequest to Node B over gRPC.
  Note: File: Node C/clients.py. Function: NodeBDenseClient.dispatch().
2. The request includes the Node A address so Node B knows where to forward the dense results.
  Note: File: Node C/clients.py. Function: NodeBDenseClient.dispatch(); config comes from Node C/config.yaml.
3. Node B immediately returns DenseDispatchAck.
  Note: File: node_B/implementation/src/server.py. Function: DenseDispatcherServicer.Dispatch().
4. Node C does not wait for dense retrieval to finish before continuing.
  Note: File: Node C/app.py. Function: handle_query() in the parallel pipeline.

This is the core latency-hiding trick in the parallel version.

6.4 Dense retrieval on Node B
1. Node B receives the dispatch request.
  Note: File: node_B/implementation/src/server.py. Function: DenseDispatcherServicer.Dispatch().
2. Node B computes a BGE-M3 embedding for the query.
  Note: File: node_B/implementation/src/server.py. Function: _async_retrieve_and_forward().
3. Node B searches Qdrant for the top-k dense matches.
  Note: File: node_B/implementation/src/server.py. Function: _async_retrieve_and_forward().
4. Node B converts the Qdrant results into RetrievedDocument messages.
  Note: File: node_B/implementation/src/server.py. Function: _async_retrieve_and_forward() and coordination_pb2.RetrievedDocument.
5. Node B measures t_dense_ms.
  Note: File: node_B/implementation/src/server.py. Function: _async_retrieve_and_forward().
6. Node B forwards the dense results to Node A with ForwardDenseResults().
  Note: File: node_B/implementation/src/server.py. Function: forward_dense_results_to_node_a().

6.5 Context fusion and generation on Node A
1. Node A receives sparse docs from Node C on the bidi stream.
  Note: File: node_A/implementation/src/main.py. Function: GenerationOrchestratorServicer.GenerateStream().
2. Node A receives dense docs from Node B via the forwarding RPC.
  Note: File: node_A/implementation/src/main.py. Function: ResultForwarderServicer.ForwardDenseResults().
3. Node A waits briefly for dense results if they have not arrived yet.
  Note: File: node_A/implementation/src/main.py. Function: _wait_for_dense_docs().
4. Node A fuses sparse and dense rankings with RRF.
  Note: File: node_A/implementation/src/main.py. Function: reciprocal_rank_fusion() in src/fusion.py.
5. Node A fetches the full text for the fused doc_ids from SQLite.
  Note: File: node_A/implementation/src/db.py. Function: get_document_texts().
6. Node A builds a prompt containing the fused context and the query.
  Note: File: node_A/implementation/src/main.py. Function: GenerationOrchestratorServicer.GenerateStream().
7. Node A calls vLLM to generate the answer.
  Note: File: node_A/implementation/src/main.py. Function: the generation path inside GenerateStream().
8. Node A streams tokens back to Node C through the gRPC stream.
  Note: File: node_A/implementation/src/main.py. Function: GenerationOrchestratorServicer.GenerateStream().

6.6 Final output to the user
1. Node C reads the token stream from Node A.
  Note: File: Node C/clients.py and Node C/app.py. Function: NodeAStreamClient and handle_query().
2. Node C forwards the streamed text to the HTTP client.
  Note: File: Node C/app.py. Function: query_endpoint() and handle_query().
3. The first token arrival time at Node C is TTFT.
  Note: File: Node C/app.py. Function: LatencyRecorder.finish() and streaming response handling.
4. The entire response time is t_total_ms.
  Note: File: Node C/app.py. Function: LatencyRecorder.finish().

7. Parallel versus sequential behavior

7.1 Parallel version
- Node C starts BM25 and dense dispatch in parallel.
- Dense retrieval and sparse retrieval overlap.
- Node A can begin generation as soon as it has the necessary context.
- This reduces TTFT by hiding local retrieval behind network and retrieval overlap.

7.2 Sequential baseline
- Node C waits for BM25 to finish.
- Node C then dispatches dense retrieval to Node B.
- Node C then continues with the Node A generation flow.
- This version exists only as a benchmark control.

The sequential baseline is used to estimate T_sequential.
The optimized version is used to measure the improvement and compute the reduction percentage.

8. Benchmarking and latency measurement

The root-level benchmark scripts compare the two Node C modes.

Benchmark objective
- Measure TTFT for the sequential baseline.
- Measure TTFT for the parallel version.
- Compare the mean or median TTFT values.
- Compute the percentage reduction.

Latency fields
- t_sparse_ms: time spent on BM25 sparse retrieval at Node C
- t_dense_ms: time spent on dense retrieval at Node B
- ttft_ms: time to first streamed token at Node C
- t_total_ms: total end-to-end response time

The benchmark files at the root are used for this comparison:
- benchmark_parallel_vs_sequential.py
- benchmark_serial_vs_parallel.py

9. Why the architecture works

The project works because each node owns a different latency domain.
- Node C handles low-cost lexical retrieval locally.
- Node B handles compute-heavy dense retrieval off the edge.
- Node A handles generation and context fusion at the core.

The system minimizes idle time by overlapping work.
- Sparse retrieval is local and fast.
- Dense retrieval is dispatched immediately and runs independently.
- Generation starts once the fused context is available.

The design also keeps data movement explicit.
- Sparse docs and dense docs are represented in protobuf messages.
- Node IDs and query IDs correlate the work across machines.
- Node A is the only place where fusion and generation meet.

11. Important implementation details

- Node B no longer behaves like a synchronous retrieval service.
- Node C should not expect dense results back from Node B directly.
- Node B forwards dense results onward instead of returning them to Node C.
- Node A is responsible for waiting on the final fused context.
- The Node A WireGuard IP is 10.8.0.1.
- The Node B WireGuard IP is 10.8.0.5.
- The Node C WireGuard IP is 10.8.0.2.
- The system uses gRPC for inter-node traffic and FastAPI only for the public edge gateway.

12. Practical summary

If you send a query to Node C:
- Node C searches BM25 locally.
- Node C dispatches dense work to Node B.
- Node B computes and forwards dense results to Node A.
- Node A fuses context and generates the answer.
- Node C streams the answer to the user.

That is the complete system.

The parallel version exists to reduce TTFT by hiding retrieval latency.
The sequential version exists to provide a baseline for fair benchmarking.

13. Answers to the first-read questions

What is gRPC?
- gRPC is a remote procedure call framework that lets one program call a function on another machine as if it were a local function.
- This project uses gRPC so Node C can dispatch work to Node B and Node B can forward results to Node A without using HTTP for every hop.
- In the code, gRPC is used for small, structured messages where the request and response types are defined in protobuf.

What is BM25 and how does it work?
- BM25 is a lexical ranking algorithm used in search engines and retrieval systems.
- It scores documents by matching query terms against document terms, rewarding exact matches while normalizing for document length.
- In practice, BM25 gives higher scores to documents that contain the query words more often, but not so much that very long documents dominate unfairly.

Explain this line: "Hosts the local Tantivy BM25 index."
- This means Node C stores the search index on its own machine instead of asking another node to do the sparse search.
- Tantivy is the search engine library that stores and queries the BM25 index.
- So when a query arrives, Node C can run sparse retrieval immediately from local disk and memory.

What is BGE-M3?
- BGE-M3 is a dense embedding model used to turn text into vectors.
- Those vectors are designed so semantically similar texts end up close together in vector space.
- Node B uses BGE-M3 to embed the query before searching Qdrant for semantically similar passages.

What is Qdrant and how is it used?
- Qdrant is a vector database.
- It stores embeddings for documents and supports nearest-neighbor search over those vectors.
- In this project, Node B sends the query embedding to Qdrant, Qdrant returns the most similar stored vectors, and Node B turns those matches into dense retrieval results.

What is "Protocol: gRPC unary dispatch"?
- "Unary" means one request goes in and one response comes back.
- "Dispatch" means the request is telling another service to start work, not to return the full final answer immediately.
- Here, Node C sends a DenseDispatchRequest to Node B, Node B acknowledges it, and Node B finishes the retrieval in the background.

What is protobuf and what is its use?
- Protobuf, short for Protocol Buffers, is a compact schema language for defining structured messages.
- It is used here to define request and response messages for gRPC calls.
- The generated Python code gives all three nodes a shared, strongly typed contract for fields like query_id, doc_id, score, and rank.

14. Expanded data model explanation

Sparse retrieval documents
- Node C BM25 returns ranked documents containing doc_id, text, score, and rank.
- doc_id is the identifier for the passage or document.
- text is the content that matched the query.
- score is the BM25 relevance value.
- rank is the position in the returned sparse list.
- These documents are wrapped into the SparseContextRequest docs field so Node A can use them later.

Dense retrieval documents
- Node B Qdrant results are converted into RetrievedDocument messages.
- doc_id identifies the stored passage.
- text carries the passage content or a reconstructed snippet.
- score is the dense similarity score from vector search.
- rank is the order of the dense matches.
- Node B forwards these documents to Node A instead of sending them back to Node C.

Final fused context
- Node A collects sparse docs from Node C.
- Node A collects dense docs forwarded by Node B.
- Node A uses Reciprocal Rank Fusion to combine the rankings from the sparse and dense lists.
- RRF is useful because it can blend two different retrieval methods without needing them to use the same score scale.
- Node A reads the full text for the fused doc_ids from its local SQLite corpus because the ranked messages only need lightweight identifiers and snippets.
- Node A then builds a prompt from the fused context and the user query before generating the answer.

15. Step-by-step notes for the flow

The numbered steps above happen in this order:
- Node C receives the query and starts timing.
- Node C runs BM25 locally.
- Node C dispatches dense retrieval to Node B over gRPC.
- Node B performs dense retrieval and forwards the result to Node A.
- Node A waits for both sparse and dense context, then fuses and generates.
- Node C streams the final output to the user and records TTFT.

The main implementation files involved are:
- Node C/app.py
- Node C/clients.py
- Node C/bm25.py
- node_B/implementation/src/server.py
- node_A/implementation/src/main.py
- node_A/implementation/src/db.py
