# Geo-Distributed RAG (2-Node Hybrid Architecture)

This project implements a two-node Retrieval-Augmented Generation (RAG) pipeline with hybrid retrieval. Node B performs both sparse (BM25) and dense (BGE-M3) retrieval concurrently, fuses results with Reciprocal Rank Fusion, and forwards fused context to Node A for LLM generation.

## Architecture (high level)

- **Node B** (hybrid retrieval gateway): FastAPI gateway, Tantivy BM25, BGE-M3 embeddings, Qdrant vector search, RRF fusion, dispatches fused context to Node A.
- **Node A** (generation): receives pre-fused doc_ids from Node B, hydrates text from SQLite corpus, generates with vLLM, streams tokens back to Node B.

**Migrated from 3-node to 2-node:** Node C (edge orchestrator + BM25) has been deprecated. Its responsibilities (BM25 retrieval, orchestration, gateway) are now handled by Node B.

## Repository layout

- `node_B/implementation/` — hybrid retrieval server, BM25 + dense retrieval, RRF fusion, Qdrant indexing.
- `node_B/docker-compose.yml` — Qdrant vector database container.
- `node_A/implementation/` — generation service, vLLM, SQLite corpus reader.
- `Node C/` — *Deprecated.* Archived from 3-node architecture.

## Prerequisites

- Python 3.10+ on all nodes
- Qdrant running for Node B (see docker-compose below)
- GPU capable of running vLLM on Node A
- SQLite corpus database for Node A (auto-generated by seed script)
- Tantivy BM25 index for Node B (auto-generated by seed script)
- Docker for Qdrant container

## Quick Start

### 1) Start Qdrant on Node B

```powershell
cd node_B
docker compose up -d
```

### 2) Install dependencies

```powershell
# Node B
cd node_B/implementation
.venv\Scripts\activate
pip install -r requirements.txt

# Node A
cd node_A/implementation
.venv\Scripts\activate
pip install -r requirements.txt
```

### 3) Seed toy dataset (100 docs)

```powershell
# From node_B/implementation (with venv activated)
python scripts/seed_toy_data.py
```

This generates:
- `node_A/implementation/corpus.sqlite` — SQLite corpus for text hydration (Node A)
- `node_B/implementation/data/tantivy_index` — BM25 index for sparse retrieval (Node B)
- Qdrant `toy_test` collection — dense embeddings for vector search (Node B)

### 4) Start Node A (generation + gRPC)

```powershell
cd node_A\implementation
python -m src.main
```

### 5) Start Node B (hybrid retrieval gateway)

```powershell
cd node_B\implementation
python -m uvicorn src.server:app --host 0.0.0.0 --port 8000
```

### 6) Query the hybrid gateway

```powershell
$body = @{ query = "what is BERT"; top_k = 10 } | ConvertTo-Json
Invoke-WebRequest -Uri "http://localhost:8000/query" -Method POST -Body $body -ContentType "application/json"
```

## APIs

- Node B: `POST /query` (JSON `{ query, top_k }`)
  - Optional header `X-Simulate-WAN-Delay` (milliseconds)
- Node B: `GET /health`
- Node A: `POST /generate` (legacy, accepts `{ fused_doc_ids, query }`)

## gRPC contracts

- Node B -> Node A: `GenerationOrchestrator.GenerateStream` (hybrid_coordination.proto)
  - Sends `HybridContextRequest` with pre-fused document list
  - Receives streaming `GenerationToken` responses

Proto file: `node_B/implementation/proto/hybrid_coordination.proto`

## Migration Notes (3-Node → 2-Node)

**What changed:**
- Node C (edge orchestrator + BM25) is deprecated.
- BM25 retrieval moved from Node C → Node B.
- RRF fusion now happens on Node B (was on Node A).
- Node B now acts as the FastAPI gateway.
- Node A receives pre-fused doc_ids and only handles text hydration + generation.
- `query_state` synchronization and `ResultForwarder` pattern removed from Node A.
- Single proto file (`hybrid_coordination.proto`) replaces the old 3-proto design.

**What stayed the same:**
- Node A still uses vLLM for generation.
- Node A still hydrates text from `corpus.sqlite`.
- Qdrant still handles dense vector search.
- WireGuard network configs are untouched.

## Notes
